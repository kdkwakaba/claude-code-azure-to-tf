# Azure 固有のプラクティス

## ワークフロールール

- `terraform plan` / `terraform apply` は実行前に**必ずユーザーに確認する**
- `terraform plan` には `ARM_SUBSCRIPTION_ID` 環境変数が必要（プロバイダーブロックへのハードコードは禁止）
- `terraform validate` / `terraform fmt` は確認なしで実行可能
- `**/*.tfstate` は読み取り専用として扱う（変更は Terraform CLI または HCL 経由のみ）
- `.terraform/**` は読み取り専用として扱う（取得済みモジュール・プロバイダー）
- `.terraform-planning-files/` が存在する場合は必ず参照する（移行・実装計画情報が含まれる可能性あり）

## ID 管理・認証

- サービスプリンシパルよりマネージド ID を優先する
- Key Vault に適切な RBAC を設定する
- 最小権限の原則を遵守する

## シークレット管理

- シークレットは Key Vault に保存する
- Terraform v1.11+ の `ephemeral` リソースや write-only パラメータを活用してステートへの書き込みを避ける
- `.tf` や `.tfstate` ファイルにシークレットを保存しない

## ネットワーク

- 新規ネットワークリソース作成前に既存の VNet／サブネット ID を確認する（ハブ＆スポーク構成の考慮）
- PaaS サービスにはプライベートエンドポイントを実装する
- リソースファイアウォールでパブリックアクセスを制限する（必要な場合はコメントで理由を明記）
- NSG および ASG を適切に設定する

## ステート管理

- ロック付きのリモートバックエンド（Azure Storage）を使用する
- 保存時および転送時の暗号化を有効にする
- ステートファイルをソース管理にコミットしない

## リソースグループ

- 指定がある場合は既存のリソースグループを使用する
- 新規作成前にユーザーに確認する

## タグ付け

既存リソースをインポートする場合は既存のタグ構造をそのまま複製し、`ManagedBy` など新しいタグキーは追加しない。新規リソースを作成する場合は `locals` で共通タグを定義してすべてのリソースに適用する。

## コスト管理

- 高額なリソースのプロビジョニング前に予算承認を確認する
- 環境に適したサイズを選定する（dev は prod より小さいサイズ）

## セキュリティ

- 監査用の診断設定を有効にする
- 過度に緩い RBAC やネットワークルールを避ける
- セキュリティ機能を利便性のために無効化しない

## アンチパターン

- パラメータ化すべき値のハードコーディング
- `terraform import` を通常ワークフローとして多用すること
- 可読性を損なう複雑な条件ロジック
- `local-exec` provisioner の不必要な使用
- 手動での Terraform 管理リソースの変更
- ローカルマシンからの本番環境への `terraform apply`
- 環境ごとにブランチ・リポジトリ・フォルダーを分けること
